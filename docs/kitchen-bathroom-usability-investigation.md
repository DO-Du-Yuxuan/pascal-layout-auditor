# 厨房与卫生间基本使用检查（V0.1）

## 数据来源与边界

本阶段只读取 Evaluation Handoff 已稳定提供的对象语义、真实尺寸、位置、旋转、Room Region、Door Portal、Door Operation、障碍物和Room Navigable Space。Parser核心未修改。

对象语义优先使用Pascal原始 `asset.category`、`asset.functionTags`、`asset.tags`、`asset.id` 和名称：

- 同义类别统一为评价器内部语义，但报告继续保留标准化ID、Pascal ID和源category。
- `category=columns` 但明确带 `sinks` 标签的对象按中等置信度水槽/洗手盆使用，并保留类别冲突诊断。
- 不读取模型外观或floor-plan图片推断设备类型。
- 设备自身红线方向（解析后的局部 `+Z`）作为前方使用区方向，与现有家具朝向定义一致。
- 冰箱、洗碗机、烤箱、淋浴门或柜门必须有独立门扇尺寸、铰链和开向才生成开启区；不得用设备整体矩形代替门扇。

## Bellevue语义调查

可靠Room：

- 厨房Room 2个：一层开放式 `OPEN KITCHEN` 所在复合Room、独立 `CHINESE KITCHEN` Room。
- 卫生间Room 4个：`BATH 1`、`MASTER BATH`、`BATH 2`、`BATH 3`。

厨房对象：

- 灶具2个，冰箱1个。
- Counter/Island语义对象3个，厨房基础柜体2个。
- 数据中另有2个 `category=sinks` 的Kitchen资产，但其几何分别匹配到非厨房Room，不能为了补齐厨房设备而强行归入厨房。
- 因此两个厨房Room内均未可靠识别水槽，Chinese Kitchen也没有冰箱。
- 没有可靠识别独立Dishwasher或Oven。
- Big Fridge没有独立门扇宽度、铰链、开向和开启角度。

卫生间对象：

- 坐便器4个，洗手盆4个。
- 淋浴房5个，浴缸2个；另有4个Showerhead作为淋浴语义辅助对象，不把花洒自身当作淋浴入口。
- 4个卫生间均有可靠Door Portal。其中MASTER BATH使用Pocket Door，不套用平开门扫掠或被困判断。
- 淋浴房没有独立淋浴门操作几何，所以G3-036只检查几何入口，不猜测淋浴门型。

## Fixture / Appliance Use Zone

统一模型生成：

- 水槽、灶具、冰箱前600毫米基本站立区；
- Counter、Cabinet、Island前600毫米操作区；
- 坐便器前600毫米坐下和起身区；
- 洗手盆前600毫米站立区；
- 淋浴房和浴缸前600毫米几何入口区；
- 有可靠设备门数据时可扩展的设备开启和操作区。

使用区至少80%位于所属Room且未被实体障碍占用，并且能从Room入口自由空间到达，才认为基本可用。Room路径沿用0.60米基本净宽、0.30米等效人员半径和0.10米导航栅格。

这些都是产品V0.1的基本使用参数，不是建筑规范、无障碍或舒适度标准。更高的厨房操作间距、卫生设备规范净空和便利性进入G2或S1。

## 重复问题归并

- G3-025发现设备接近问题时引用G3-026，不重复生成相同问题卡；数据语义缺失仍由G3-025说明。
- G3-030若与G3-002/G3-003/G3-006为同一入口根因，只保留基础通行问题。
- G3-038若只是某一个洁具使用区被占用，引用G3-034/G3-035/G3-036；只有两个核心设备形成相互完全死锁才独立报警。

## 页面验证方式

页面增加默认关闭的“显示厨卫使用区”开关。打开后，绿色表示可用使用区，红色表示被阻挡，黄色虚线表示数据待确认。问题卡继续复用“在图中查看”、自动切换楼层、红色主要对象、橙色关联对象和折叠技术详情。

G3-025缺少厨房设备语义时，优先高亮对应的厨房Zone（例如 `OPEN KITCHEN` 或 `CHINESE KITCHEN`），并将已识别设备作为橙色关联对象；只有缺少可用厨房Zone轮廓时才退回物理Room Region，避免开放式客餐厨把整片公共Room误显示为厨房问题范围。

按用户之前的验收约定，本阶段不使用浏览器代替人工画布检查；Bellevue的Chinese Kitchen Stove应由用户在画布上核对其红线朝向、墙体关系和600毫米使用区。
