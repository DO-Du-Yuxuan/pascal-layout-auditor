# 卧室与客餐厅家具基本使用检查（V0.1）

## 数据语义调查

评价器读取 Evaluation Handoff 中保留的 `assetId`、`assetName`、`category`、`functionTags`、`assetTags`、真实尺寸、位置和旋转。Parser 核心不参与家具语义判断。

Bellevue 中能够由资产字段可靠识别：

- Bed语义对象：5件。其中4张是双人床：2张具有可靠 `double-beds` 类别，另2张虽源 category 错标为 `columns`，但明确的 `double-beds` functionTags 足以按中等置信度双人床评价，同时保留源类别冲突诊断。另有1件名为Bench的对象因源 category=`single-beds` 被数据明确标成单人床，本轮按源语义参与评价。
- Wardrobe/Closet：21 个，家具类别可靠；但JSON没有门型或开启方向，因此本阶段取消衣柜开启判断，不产生无法判断。
- Sofa：2 件；Armchair/Lounge Chair：3 件。画布“家具中心”红线使用解析后的家具局部 `+Z` 轴，可直接作为坐具正面方向。
- Dining Table：2 张；Dining Chair：8 把。餐椅方向直接来自每把椅子的JSON rotation，不需要先匹配餐桌。
- Coffee Table：4 张；Desk：6 张；TV Cabinet：0 个。
- 房门、窗户及卫生间入口具有几何和宿主关系。窗户所属Room原则上可由宿主墙、Window位置和Room Region派生；当前缺少的是窗户用途语义，而不是空间归属。
- JSON 没有项目计划用餐人数或餐桌计划座位数。

不能根据模型外观、floor-plan 图片或矩形包围盒猜测衣柜门型、计划人数和必要窗户。坐具正面不再猜测，而是与画布共同使用JSON rotation解析后的局部 `+Z` 轴。

## Furniture Use Zone

Furniture Use Zone 位于评价器几何与关系层，所有专项规则复用同一结果：

- 床左右侧上下床区域：每侧深 0.60 m；至少一侧可用即满足本轮基本使用。双人床只有一侧可用仍通过，同时保留便利性 diagnostic。
- 衣柜开启区域：本阶段取消，不生成使用区、不产生问题或无法判断。
- 沙发和主要坐具进入区域：沿家具中心红线方向（局部 `+Z`）深 0.45 m。
- 餐椅拉出区域：家具中心红线为前方（局部 `+Z`），沿红线反方向（局部 `-Z`）深 0.60 m；餐桌不参与方向推断。
- 使用区至少 80% 位于所属 Room 且未被其他实体 footprint 占用，才认为基本可用。

以上均为产品 V0.1 基本使用参数，不是建筑规范、无障碍或舒适度标准。家具较紧但仍能基本使用的情况留给 S1。

家具匹配 Room 时使用家具旋转 footprint 与 Room Region 的相交面积，不使用对象中心点。家具没有任意形状碰撞轮廓时，使用真实尺寸和旋转生成方向矩形；因此异形家具仍存在模型精度限制。

## 规则边界与去重

- G3-014 检查床 footprint 是否完整位于可靠 Room；当前两个确定问题同时可追溯到已有 G1-023 越界/嵌墙源数据问题。
- G3-015 至少一侧床边使用区可用即通过；低置信床不猜测。明确 `double-beds` functionTags 与错误 category 冲突时按中等置信度评价，并保留诊断。
- G3-016 只对可靠或具有明确双人床标签的中等置信度双人床执行；一侧可用不判失败。
- G3-017 和 G3-021 复用 Room Navigable Space。若同一阻断已由 G3-003/G3-004/G3-006报告，专项规则返回引用 diagnostic，不重复生成问题卡。
- G3-018 本阶段停用，固定返回 `not_applicable`，不再让21个衣柜产生无法判断。
- G3-019 正式限定为“床不得阻断卧室房门、套内卫生间入口及已建立的主要通行路径”。衣柜操作由G3-018独立处理；普通窗户接近和开启留给G3-013；紧急逃生窗属于G2。缺少窗户用途语义不再导致本规则无法判断。
- G3-020 使用家具中心红线方向生成坐具前方基本进入区。
- G3-022 可以统计餐椅，但缺计划人数时无法判断“与计划人数相符”。
- G3-023 直接使用餐椅自身朝向：红线反方向即后方拉出方向，不依赖餐桌位置。
- G3-024 只对同一 Room 内同时存在客厅和餐厅 Zone 的复合空间执行。
